零改动这三个 chip 的成本

ActivityListPage 已经收 username + filter + title(AppRoutes.ets:128), 现在由调用方写死成四个割裂入口(ProfilePage.ets:280/330/532/544)。

加 chips 不是新功能,是把已有参数暴露出来。顺带好处:「我的」页那三个入口(我的话题/我的回复/我的点赞)也能合并到同一页,少三个跳转。

没做「赞」这个 chip

filter=1 成本同样为零,但互动圈已经有「赞过」那一行了。同一件事在一个页面里出现两次,是臃肿而不是丰富。

要加随时能加——一行常量。但默认不加。

砍掉了什么

这一节比上面两节重要。清爽是砍出来的,不是排出来的。

官网有为什么不做
Boosts独立插件接口 /discourse-boosts/users/{u}/boosts-given.json,响应结构自成一套(带 raw/cooked 短评)。「他给谁 boost 过」对判断这个人几乎零贡献,却要一套映射 + 一套气泡渲染
回应独立接口 /discourse-reactions/posts/reactions.json,返回裸数组,还要按 reaction_value 查 emoji 出图。同理:「他对什么帖发过 🤗」不构成认知
投票独立接口 /topics/voted-by/{u}.json。这个反而最省(标准 topic_list,复用 TopicListItem),但「他投过哪些票」信息价值最低——省不代表该做
等级进度connect.linux.do 渲染的永远是 SSO 当前登录者,技术上拿不到别人的。所以信任级别药丸不可点,只是标签
已读 / 书签 / 草稿/read.json 固定返回自己;书签私有(取他人 403);草稿是私有物。技术上不可能
设置 / 退出登录他人视角下无意义。这也是不改造 987 行 ProfilePage 而新建页的原因——那里面混着登出/设置/草稿/未登录态
常逛分类数据白送(summary.json 的 top_categories),但它排在第七位才出现,读者早已划走。白送也不等于该放
作品集 / 关注列表官网的两个 tab。移动端一屏就那么大,优先级排不上

关键设计判断

核心五个区块,两个请求

官网那一大堆区块(热门回复/热门话题/最多回复至/被谁赞得最多/赞最多/热门类别)全部来自同一个 summary.json —— 你已经在请求它了(getSummary),只是 DiscourseUserSummary(UserProfileService.ets:41-50)只声明了 stats 和 badges,其余字段 JSON.parse 后被静默丢弃。

所以整页的信息量,成本是扩展映射,不是加请求。detail + summary 两个请求渲染完两屏,「最近」段延迟到切换时再拉。

唯一例外为什么只留「解决方案」

Boosts / 回应 / 投票 / 已解决 都是独立插件接口、都要一套新映射。前三个我全砍了,只留这个,因为它是四个里唯一回答「他靠不靠谱」的 —— 被采纳的答案是能力的硬证据,正是点开陌生人头像时最想知道的事。

而且它有天然入口:统计格里本来就有「解决方案」这个数字(solved_count)。让唯一有分量的数字可点,而不是新开一个 chip。接口还自带 offset/limit 分页,列表页照 ActivityListPage 抄即可。

主张互动圈:三张长列表压成三行

「最多回复至 / 被谁赞得最多 / 赞最多」是 Discourse 最有辨识度的部分,回答的是社交问题:这人跟谁混、在哪个圈子。官网用三个各占半屏的列表铺开,手机上照抄要滚两屏。

压成三行头像后信息密度反而更高——一眼扫完三种关系。次数挂头像右下角小角标,顺序本身已表达强弱。头像可点,递归进本页。

取舍为什么用分段而不是平铺

热门话题 / 热门回复 / 最近 都是「话题行」列表,视觉几乎一样。平铺三个区块会让页面读起来像同一段重复三遍,还多滚一屏半。

做成分段后三者共用一块区域,标题栏即切换器——移动端处理同构列表的标准解法。实现注意:切换靠 @State 驱动;别用 @Builder 按值传参渲染分段内容,那玩意不随 @State 重绘。

陷阱头像点击的冒泡冲突

本 SDK 没有 stopPropagation。TopicListItem、ActivityListPage、BookmarksPage 的整行都已挂 onClick 跳帖子详情——头像上再挂一个,两者会同时触发。

解法沿用 ProfilePage.ets:176-177 已经踩过并写在注释里的做法:外层 Row 不挂 onClick,逐个子元素分别挂。这是本次最容易翻车的地方,涉及 10 个头像调用点。

复用数据层几乎不用动

UserProfileService 五个方法全部已按 username 为首参,没有「当前用户」隐含语义;UserProfileCache 已按 username 分键,不会串数据。

BadgesPage、ActivityListPage 也已参数化,传目标 username 就能用,只需把标题从「我的话题」改成「TA 的话题」。

附:三个被砍页的行模板(要捡随时捡)

你抄的四个接口没白费——它们证明了这三个页怎么做,同时也让我看清它们不该做。 行模板画在这儿,哪天你改主意,照着实现即可。三个页各自的成本都写清楚了。
